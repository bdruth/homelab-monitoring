**Symptom (diagnosed 2026-06-14):** homelab-monitoring Actions runs "hang ~15 min then fail in the complete step with no logs." Jobs actually SUCCEED (~3 min) — all steps pass, runner logs `🏁 Job succeeded`, and the compressed log IS stored (`action_task.log_in_storage=t`, `.log.zst` on disk).

**Root cause:** The task-finalizing `UpdateLog` call (`Rows:[], NoMore:true`) returns **HTTP 500** (fast, ~60-100ms — not a timeout; no Postgres error). Gitea **1.26.2** PR #37687 made that finalize call run `TransferLogs` inline, and it intermittently 500s after the transfer already succeeded. Because the runner never gets a 200 on finalize, the task is never marked complete, so Gitea's **zombie-task reaper** (`clear_tasks.go stopTasks()`) force-marks it `failure` ~15 min later (`action_task` dur ≈ 900s vs ~138s for a real success). The `[W] Cannot transfer logs ... dbfs.Open ... file does not exist` warning is a benign side effect (log already moved). Intermittent — some runs finalize cleanly and succeed.

**Exact upstream issue: [#37871](https://github.com/go-gitea/gitea/issues/37871)**, fixed by [#37885](https://github.com/go-gitea/gitea/pull/37885) ("ack re-sent UpdateLog finalize idempotently"), backported to `release/v1.26` as [#37892](https://github.com/go-gitea/gitea/pull/37892). Mechanism precisely: 1.26.2 (#37687) added TransferLogs-on-finalize but rejected any RE-SENT finalize with `AlreadyExists` → HTTP error → runner never gets a 200 ack → task reaped. The fix returns an idempotent success ack on re-send. **The fix is NOT in the v1.26.2 tag** — it's queued on `release/v1.26` for **1.26.3** (21 commits pending as of 2026-06-14). 1.26.1 works because it predates the TransferLogs-on-finalize change.
